# Memory Promotion Protocol

用途：定义记忆如何从日常观察晋升为长期偏好、规则、公理或技能。  
目标：减少记录很多，但沉淀很少的上下文腐烂。

## 一、四层结构

1. L1 `OBSERVATIONS.md`
   - 记录事实、决策、证据、错误与候选模式
2. L2 `WEEKLY_REFLECTIONS.md`（使用 `reflector_weekly_template.md` 生成）
   - 每周识别重复模式、筛掉噪音、提出晋升候选，并保留周级证据链
3. L2.5 `PREFERENCE_REGISTRY.md`
   - 存放稳定但仍可调整的长期偏好
4. L3 `rules/` / `axioms/` / `skills/`
   - 存放已经足够稳定、需要直接改变默认行为的内容

## 二、晋升标准

### 从观察到周模式

满足任一条件即可进入周反思：

- 一周内重复出现两次以上
- 导致重要返工或重要节省时间
- 暴露出明显的审稿风险、识别风险、写作风险或工作流风险

### 从周模式到长期偏好

一般需要同时满足：

1. 至少有两个独立证据点或项目实例
2. 能写成一句明确的默认偏好
3. 能写清适用条件和失效边界
4. 未来一个月内大概率还会复用

### 从长期偏好到 rules

满足以下情形之一时晋升到 `rules/`：

- 已经成为默认行为约束
- 不遵守它会持续造成高成本错误
- 它需要在每次 session 启动时被动加载

### 从长期偏好到 axioms

满足以下情形之一时晋升到 `axioms/`：

- 它不是局部技巧，而是跨任务的判断原则
- 它带有强烈的为什么这么做的哲学底层
- 它在未来半年都很可能稳定成立

### 从长期偏好到 skills

满足以下情形之一时晋升到 `skills/`：

- 它已经不只是原则，而是可重复执行的流程
- 它有清晰触发条件、步骤和交付格式
- 它适合作为怎么做的标准操作单元

## 三、降级与退休

以下情况应降级或退休：

- 只在单一项目中成立
- 已被更高层规则覆盖
- 新证据证明它只是一时噪音或过拟合习惯

降级顺序通常为：

`rules/axioms/skills` -> `PREFERENCE_REGISTRY.md` -> `WEEKLY_REFLECTIONS.md` -> `OBSERVATIONS.md` 历史保留

## 四、每周最小动作

每周反思至少完成四件事：

1. 从 L1 中挑出 2-5 条高价值重复信号
2. 在 `WEEKLY_REFLECTIONS.md` 中写清这些信号属于审稿、识别、写作、工作流还是协作
3. 更新或新增 1-3 条长期偏好
4. 判断是否有 0-2 条内容需要晋升到 `rules/`、`axioms/` 或 `skills/`

## 五、写偏好时避免的错误

- 只写抽象口号，不写默认动作
- 没有边界条件，导致过度泛化
- 证据不足就直接升级到规则
- 把短期项目状态误写成长期偏好

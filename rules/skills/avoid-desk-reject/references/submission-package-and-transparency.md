# Submission Package and Transparency

## Use This Module When

- The user asks about cover letters, overlap, formatting, AI disclosure, or submission readiness
- The paper may be promising but the package around it signals immaturity
- The question is whether the manuscript is ready to be sent rather than merely readable

## What Editors Are Screening For

- Whether the cover letter helps the editor understand fit, contribution, or special context
- Whether overlap with related manuscripts is disclosed clearly
- Whether formatting and journal instructions were followed
- Whether data sources, coding choices, and AI use are transparent where relevant
- Whether the package signals care, maturity, and low hidden-risk

## Common Desk-Reject Triggers

- No useful cover letter when special context needed explanation
- Hidden overlap with other manuscripts using the same data or framing
- Scope mismatch left unexplained
- Obvious noncompliance with journal format or style guide
- Opaque use of AI, data collection procedures, or coding processes
- A package that feels undercooked even if the core idea might have promise

## Questions to Answer

- What does the editor need to know before assigning the manuscript?
- Is there any overlap, conflict, or disclosure issue that should be surfaced immediately?
- Does the submission package reduce ambiguity or increase it?
- Have the journal's basic formatting and submission expectations been met?
- Does the package look like a finished scholarly submission rather than a near-draft?

## Safe Outputs for the Preflight Report

- Keep package risk separate from contribution risk; both matter, but they are not the same
- Use `major revision before submit` when transparency or maturity signals are the main blockers
- Recommend explicit disclosure actions rather than vague advice to "be more transparent"
